# Develop Feature Workflow

## 目的

機能開発を、設計判断と実装作業を混ぜずに次の順で進める。

1. 目的から TOBE を描く。
2. TOBE の不変条件と AS-IS 由来の制約を分ける。
3. ギャップと移行方針を整理する。
4. 既存コードへ最小 diff で実装する。
5. 検証とレビューを通して仕上げる。

純粋な設計依頼では、TOBE、制約、ギャップ、移行方針までを出し、ユーザーが実装を求めていない限りコード変更はしない。

## 基本原則

- AS-IS は事実として扱い、TOBE の上限として扱わない。
- 変更コストを理由に、責務境界、モデル、契約、ユーザー体験を歪めない。
- TOBE と移行計画を分ける。高コストな変更でも、TOBE として正しいなら維持し、段階移行や互換層は別に設計する。
- 実装フェーズでは既存コードの形を尊重し、要件と無関係なリファクタリング、整形、命名変更、依存更新を混ぜない。
- ユーザーの未コミット変更を上書きしない。
- 不明点は、設計または実装を誤るリスクが高いものだけ確認する。
- 低リスクな細部は合理的に仮定し、最終報告で明記する。
- 検証できていないことを完了扱いしない。

## 1. 目的と作業範囲を固定する

作業開始時に、依頼が設計だけなのか、実装まで含むのかを判断する。

- 解く課題、成功条件、利用者、守るべき品質を整理する。
- 変更すべき振る舞い、変更しない振る舞い、影響範囲を切り分ける。
- 不明点があっても TOBE を描ける範囲では作業を止めず、仮定として明記する。
- 実装まで行う場合は、`git status` で作業ツリーを確認し、ユーザーの既存変更を把握する。

必要なら短く共有する。

```text
作業前確認:
- 目的:
- 成功条件:
- 変更する振る舞い:
- 変更しない範囲:
- 作業範囲: 設計のみ / 実装まで
- 確認が必要な点:
```

## 2. TOBE を先に描く

既存実装、現行業務、移行コスト、目先の変更量へ早く寄せず、目的からあるべき姿を描く。

- ドメインモデル、責務境界、データフロー、外部契約、ユーザー体験、運用責務を目的から設計する。
- 既存のファイル構成、クラス、テーブル、API の形へ早く寄せない。
- 複数案がある場合は、目的への適合度で比較し、変更量の少なさだけで選ばない。
- TOBE で守るべき不変条件を短く列挙する。

設計の出力が必要な場合は、次の型を使う。

```markdown
## 目的
- 

## TOBE
- 

## 不変条件
- 

## AS-IS 由来の事実と制約
- 恒久制約:
- 暫定制約:
- 偶発的制約:

## ギャップと移行
- 
```

## 3. AS-IS と移行方針を分ける

TOBE を決めた後で、実装対象の現状を読む。

- `AGENTS.md`、README、関連テスト、既存実装を読む。
- 現行実装、依存関係、移行リスクを事実として整理する。
- 既存コードや現行運用に由来する制約を、恒久制約、暫定制約、偶発的制約に分類する。
- AS-IS との差分、変更コスト、リスク、段階移行、互換性維持を整理する。
- 移行都合で TOBE そのものを弱めず、実装順序や互換層で扱う。

## 4. 最小 diff の実装方針を決める

TOBE と移行方針を踏まえて、今回入れる実装範囲を絞る。

- 触る予定のファイルを短く列挙する。
- 既存の責務境界、命名、テストスタイルに合わせる。
- まず user-visible behavior または contract に必要な変更だけ入れる。
- ついでの cleanup は、今回の変更を成立させる場合だけ行う。
- 既存挙動を変える場合は、その理由が TOBE または要件と結びついていることを確認する。
- テストはリスクに見合う最小単位で追加または更新する。

実装前に、必要なら短く共有する。

```text
実装方針:
- TOBE:
- 不変条件:
- 今回実装する範囲:
- 今回実装しない範囲:
- 触る予定のファイル:
- 最小 diff 方針:
- 検証方針:
```

避けるもの:

- TOBE を固める前の場当たり的な実装
- 横断的な書き換え
- 先回りした汎用化
- 使われていない helper 追加
- 大量の snapshot 更新
- 要件と関係ない型、lint、format の大掃除

## 5. 実装する

要件を満たす最短経路で編集する。

- 既存パターンに沿って変更する。
- 変更対象外の未コミット差分を巻き込まない。
- 実装中に TOBE と矛盾する事実が見つかった場合は、TOBE、移行方針、実装範囲のどれを見直すべきか切り分ける。
- 大きな方針変更が必要になった場合は、ユーザーに確認する。

## 6. フィードバックを判定する

ユーザーが「こっちのほうがいいのでは」「この方が早いのでは」と提案した場合、すぐに取り込まず、次を確認する。

- その提案は、設計目的や成功条件をよりよく満たすか。
- 新しい事実、制約、リスクを示しているか。
- TOBE の不変条件と矛盾しないか。
- AS-IS の都合、短期の変更コスト、個人的な好みだけを根拠にしていないか。
- 長期的な責務、拡張性、運用性、ユーザー体験を劣化させないか。

採用する場合は、どの不変条件や目的をよりよく満たすのかを説明して TOBE を更新する。採用しない場合は、理由を明確に述べ、元の TOBE を維持する。

## 7. 検証する

狭い範囲から検証する。

1. 変更した振る舞いを直接見るテスト
2. 近傍の関連テスト
3. 必要に応じて lint、typecheck、build

検証コマンドを実行できない場合は、理由と残リスクを明記する。失敗した場合は、diff を増やす前に原因を確認する。

## 8. レビューパスを通す

実装と自己確認の後、可能ならレビュアーサブエージェントにレビューさせる。

レビュー依頼は、TOBE、実装範囲、diff、検証結果に絞る。

```text
この変更をレビューしてください。
- 要件:
- TOBE:
- 不変条件:
- 今回実装した範囲:
- 今回実装しなかった範囲:
- 変更ファイル:
- 主要 diff:
- 実行した検証:
- 特に見てほしいリスク:

観点:
- TOBE と実装が矛盾していないか
- 要件を満たしているか
- diff が過剰でないか
- 既存挙動を壊していないか
- テストが不足していないか
- 命名、責務配置、エラー処理が既存パターンに合うか
```

レビュー結果の扱い:

- 明確なバグ、回帰、テスト不足は修正する。
- 好みの問題やスコープ外提案は採用しない。
- 大きな方針変更が必要な指摘は、TOBE と実装前提を整理してユーザーに確認する。
- レビューで直した後は、関係する検証を再実行する。

## 9. 最終報告

最終報告では、設計判断、変更量、検証結果が分かるように短くまとめる。

```text
Summary:
- TOBE:
- 変更:
- 触った主なファイル:
- 最小 diff のためにやらなかったこと:
- レビュー結果:
- 検証:
- 残リスク:
```

完了と言う前に、設計、実装、レビュー対応、関連検証が終わっていることを確認する。設計だけの依頼では、実装していないことを明記する。
