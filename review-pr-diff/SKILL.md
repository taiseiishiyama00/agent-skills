# PR Diff レビュー

## 概要

このスキルは、PR またはブランチ間の差分を中心にコードレビューするために使う。レビューでは重大なバグ、回帰、データ破壊、セキュリティ問題、互換性破壊、テスト不足を優先し、出力はクリティカルな順に最大3件までに絞る。加えて、動作には直ちに現れにくい不要コード、責務配置の不一致、既存設計からの逸脱、将来の負債化リスクも別枠で最大3件まで指摘する。

## 発火例

- PR `oo` をレビューして。
- この PR のコードレビューをして。
- `feature/foo` と `main` の差分をレビューして。
- `oo` ブランチと `xx` ブランチの Diff を見て。
- PR の変更に危険な点がないか確認して。

## 基本方針

- レビューではコードを変更しない。
- 評価前に比較対象の各ブランチを pull して最新化する。
- pull 前に `git status --short --branch` を確認し、未コミット変更を壊す可能性がある場合は、stash、別 worktree、作業中止などの保護方法をユーザーへ確認する。
- Diff を中心に評価する。必要な場合だけ、関連コード、テスト、履歴、設定、ドキュメントへ範囲を広げる。
- スタイル好み、軽微な命名、任意のリファクタリングは、重大な不具合または将来の負債化リスクにつながらない限り出力しない。
- 動作不具合・安全性・テスト不足の指摘は最大3件にする。3件を超える場合は、ユーザー影響と破壊度が高い順に選ぶ。
- 設計・負債化リスクの指摘も最大3件にする。3件を超える場合は、変更容易性、責務境界、削除容易性、既存パターンとの整合性への影響が大きい順に選ぶ。

## ワークフロー

1. レビュー対象を特定する。
   - PR 指定の場合は、PR の base ブランチと head ブランチを特定する。GitHub などの連携ツールや `gh pr view` が使える場合はそれを使う。
   - ブランチ指定の場合は、比較元と比較先を特定する。
   - 指定が曖昧な場合は、レビュー前に対象 PR またはブランチ名を確認する。

2. 各ブランチを最新化する。
   - pull 前に `git status --short --branch` で現在の作業ツリーを確認する。
   - 未コミット変更が安全に保護できない場合は、pull せずにユーザーへ確認する。
   - base/head または指定された2ブランチをそれぞれ pull し、リモートの最新状態を取り込む。
   - pull できない場合は、その理由を記録し、最新化できていない前提でレビューを続けてよいか確認する。

3. Diff を作る。
   - `git merge-base <base> <head>` で共通祖先を確認する。
   - 原則として `<merge-base>...<head>` の差分をレビュー対象にする。
   - ファイル一覧、変更行、削除、リネーム、テスト追加有無を確認する。
   - 大きな差分では、まず変更ファイルをリスク別に分類する。

4. Diff を中心に評価する。
   - 変更行から実行時バグ、例外、null/境界値、競合状態、状態不整合、データ破壊、権限漏れ、互換性破壊を探す。
   - API、永続データ、設定、認証/認可、外部連携、非同期処理、キャッシュ、トランザクション、エラーハンドリングの変更を優先的に見る。
   - テスト変更がない場合は、変更のリスクに対してテスト不足が重大かを判断する。
   - 変更の意図が不明な場合は、関連コミット、既存実装、呼び出し元、テストを最小限確認する。

5. 設計・負債化リスクを評価する。
   - 使われないコード、到達不能コード、過剰な設定、不要な分岐、今回の要件に不要な一般化や抽象化がないか確認する。
   - 実装場所が既存の責務境界、レイヤー、ドメインモデル、依存方向に合っているか確認する。
   - 既存の helper、service、adapter、validation、state 管理、テスト支援 API などを無視した重複実装や局所解がないか確認する。
   - 変更が将来の機能追加、削除、テスト、移行、ロールバックを不必要に難しくしないか確認する。

6. 指摘を選別する。
   - クリティカルな順に最大3件だけ出力する。
   - 設計・負債化リスクは別枠で最大3件だけ出力する。
   - 同じ根本原因の指摘は1件にまとめる。
   - 根拠が弱い推測は出さない。必要なら「確認事項」として短く扱う。
   - 問題が見つからない場合は、「重大な指摘なし」と明記する。

## 出力形式

指摘がある場合は、まず重大度順に最大3件だけ出力する。

```markdown
1. [重大度] `path/to/file.ext:line`
   問題: ...
   根拠: ...
   修正案: ...
```

重大度は次のいずれかを使う。

- `Critical`: データ破壊、セキュリティ問題、重大な本番障害、明確なクラッシュ、互換性破壊。
- `High`: 一般的なユーザー影響のあるバグ、重要機能の回帰、重要なテスト不足。
- `Medium`: 条件付きで発生する不具合、局所的な回帰、将来の保守で高リスクな問題。

設計・負債化リスクの指摘がある場合は、その後に最大3件だけ出力する。

```markdown
設計・負債化リスク:
1. `path/to/file.ext:line`
   問題: ...
   根拠: ...
   修正案: ...
```

設計・負債化リスクは、直ちに動作バグとして現れなくても、不要コード、責務配置の不一致、既存パターンからの逸脱、重複実装、過剰な抽象化、削除や変更を難しくする構造を対象にする。単なる好みや一貫性だけの指摘は出さない。

指摘がない場合は、次の形式にする。

```markdown
重大な指摘なし。
設計・負債化リスクの重大な指摘なし。

確認した範囲:
- base/head または比較ブランチ
- 主な差分領域
- 実行または確認したコマンド
```

## 完了基準

レビュー対象の両ブランチを最新化したか、最新化できない理由を明記していること。Diff を中心に確認し、クリティカルな順に最大3件の指摘、加えて設計・負債化リスクの指摘を最大3件、またはそれぞれ「重大な指摘なし」を出力していること。レビューではコードを変更しない。
