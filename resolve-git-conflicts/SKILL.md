# Git コンフリクト解決

## 概要

このスキルは、Git のマージコンフリクトを安全に解決するために使う。明らかな機械的衝突は自動で解決し、意味判断が必要な衝突は両ブランチの履歴、時系列、関連変更、テスト結果を判断材料にする。

## 発火例

- `oo` ブランチと `xx` ブランチのコンフリクトを解決して。
- PR `oo` のコンフリクトを解決して。
- この PR のマージコンフリクトを直して。
- `feature/foo` を `main` に追従させたい。
- rebase 中の衝突を解消して。

## 基本方針

- `git reset --hard`、`git checkout -- <path>`、`git clean` など、ユーザー変更を失う可能性がある破壊的操作は行わない。
- `--ours` または `--theirs` の丸ごと採用を安易に使わない。使う場合は、ファイル全体を片側採用してよい根拠を確認する。
- 既存の未コミット変更を保護する。コンフリクト解決と無関係な変更は触らない。
- 解決後に勝手に commit、push、merge continuation をしない。ただしユーザーが明示的に依頼した場合は、その依頼範囲で実行してよい。
- 判断できない衝突は推測で確定しない。選択肢、根拠、リスクを提示してユーザーに確認する。
- 評価を始める前に、対象となる両ブランチを pull して最新化する。pull によりユーザーの未コミット変更が壊れる可能性がある場合は、先に保護方法を確認する。

## ワークフロー

1. 対象を特定し、両ブランチを最新化する。
   - ユーザーが `oo` ブランチと `xx` ブランチを指定した場合は、その2ブランチを解決対象として扱う。
   - ユーザーが PR 番号、PR 名、または「この PR」と指定した場合は、PR の base ブランチと head ブランチを特定する。GitHub などの連携ツールや `gh pr view` が使える場合はそれを使う。
   - 評価前に対象の両ブランチを pull して、リモートの最新状態を取り込む。
   - pull 前に `git status --short --branch` を確認し、未コミット変更がある場合は、それらを消さない方法で進める。必要ならユーザーへ stash、別 worktree、作業中止のどれで進めるか確認する。
   - pull できないブランチ、権限不足、PR 情報が取れないなどの問題がある場合は、評価を進める前に理由と代替案を報告する。

2. 現在の Git 状態を確認する。
   - `git status --short --branch` で現在のブランチ、操作中の状態、未コミット変更を確認する。
   - `.git/MERGE_HEAD`、`.git/rebase-merge`、`.git/rebase-apply`、`.git/CHERRY_PICK_HEAD` の存在から、merge、rebase、cherry-pick などの操作種別を把握する。
   - `git diff --name-only --diff-filter=U` でコンフリクト中のファイルを列挙する。
   - コンフリクト解決と無関係な変更がある場合は、触らない前提で作業する。

3. 衝突箇所を読む。
   - 各ファイルの `<<<<<<<`、`=======`、`>>>>>>>` を確認する。
   - 近傍コードも読み、衝突箇所が API、データ構造、設定、生成物、テスト、ドキュメントのどれに関係するか把握する。
   - 生成ファイルやロックファイルの可能性がある場合は、手編集より再生成やパッケージマネージャの標準解決を優先する。

4. 両ブランチの履歴と時系列を調べる。
   - `git merge-base HEAD <other>` または現在の操作状態から共通祖先を確認する。相手ブランチ名が不明な場合は、Git 状態、reflog、MERGE_HEAD、rebase todo から推測する。
   - コンフリクトファイルごとに、両側の関連コミットを確認する。
   - 例: `git log --oneline --decorate --left-right --merge -- <path>`、`git log --oneline --follow -- <path>`、`git show <commit> -- <path>`。
   - 新しい変更かどうかだけで判断しない。コミットメッセージ、変更理由、関連ファイル、テスト追加、後続コミットでの修正を読む。

5. 解決方針を分類する。
   - 明らかな機械的衝突: import/order、フォーマット、隣接行の追加、同じ意図の rename、単純なリスト結合など。自動で解決する。
   - 意味的な統合が必要な衝突: 両側の機能追加、API 変更、条件分岐変更、データモデル変更、テスト期待値変更など。履歴と関連コードを踏まえて統合案を作る。
   - 判断不能な衝突: どちらの振る舞いを優先すべきか仕様判断が必要なもの。ユーザーへ確認する。

6. 解決を実装する。
   - 衝突マーカーを完全に取り除く。
   - 両側の意図を保つ統合を優先する。
   - 片側を捨てる場合は、捨てる理由が履歴や仕様から説明できる状態にする。
   - コンフリクトファイルだけでなく、必要に応じて関連テストや呼び出し元も最小限更新する。

7. 検証する。
   - `rg -n '<<<<<<<|=======|>>>>>>>'` などで衝突マーカーが残っていないことを確認する。
   - `git diff --check` で空白エラーや基本的な差分問題を確認する。
   - 関連するテスト、ビルド、フォーマット、lint を実行する。リポジトリの標準コマンドがある場合はそれに従う。
   - 検証失敗時は、失敗内容を読み、コンフリクト解決が原因か既存問題かを切り分ける。

8. 結果を報告する。
   - pull した対象ブランチ、PR の base/head、または最新化できなかった理由を示す。
   - 解決したファイルを列挙する。
   - 主な採用判断と、履歴・時系列から見た根拠を簡潔に説明する。
   - 実行した検証コマンドと結果を示す。
   - 未解決の判断事項、未実行の検証、残リスクがある場合だけ明記する。

## ユーザー確認が必要な場合

次の場合は、実装を確定する前にユーザーへ確認する。

- 両側の変更が互いに排他的で、どちらの仕様を優先すべきか履歴から判断できない。
- 片側の変更を捨てる必要があるが、捨ててよい根拠が弱い。
- public API、永続データ、マイグレーション、認証/認可、課金、セキュリティ、外部連携の振る舞いが変わる。
- テスト期待値の変更が、仕様変更なのかバグ修正なのか判断できない。
- コンフリクト解決のために大きなリファクタリングが必要になりそう。

確認時は、次を提示する。

- 選択肢
- 各選択肢の根拠
- 履歴と時系列から分かること
- 推奨案
- 推奨案のリスク

## 完了基準

衝突マーカーが残っておらず、Git が未解決コンフリクトを報告しておらず、関連検証を実行し、その結果を報告できる状態になるまで完了としない。検証できない場合は、理由と残リスクを明記する。
