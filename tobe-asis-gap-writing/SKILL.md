# TOBE ASIS GAP Writing

## Overview

PR、issue、バックログなどの説明文を、対象プログラムやシステムの完了後の状態から逆算して整理する。「どの状態を目指すのか」「現状はどうなっているのか」「どの差分を埋めるのか」「どう対応するのか」を追いやすい構成にする。

## 基本方針

- まず TOBE を書く。作業が完了した後に、対象プログラムやシステムがどのような状態になるべきかを示す。
- 次に ASIS を書く。現在の挙動、制約、未整備な点、困っていることを事実ベースで示す。
- GAP では、TOBE と ASIS の差分を明確にする。単なる作業項目ではなく、なぜ対応が必要かを示す。
- 対応方針では、GAP を埋めるために何をするかを書く。PR では実施済みの変更、バックログでは実施予定の作業を書く。
- 説明文は、目的、現状、差分、対応内容を短時間で把握できる量に絞る。

## 作成手順

1. 対象を判定する。
   - PR 説明または PR コメント: 実装済みの変更、差分、確認内容を中心にする。
   - issue またはバックログ: 目的、完了条件、対応予定、受け入れ条件を中心にする。
   - 作業依頼: 依頼者と実装者が同じ完了イメージを持てることを優先する。

2. TOBE を書く。
   - 「この PR / バックログが完了すると、対象プログラムやシステムがどういう状態になるか」を書く。
   - 実装方法ではなく、到達状態、実行可能な状態、検証可能な状態を書く。
   - 曖昧な表現だけで終えず、確認可能な状態に寄せる。

3. ASIS を書く。
   - 現在の挙動、運用、制約、問題、手作業を整理する。
   - 事実と推測を混ぜない。
   - 「何ができていないか」だけでなく「対象プログラムやシステム上の問題として何が起きているか」を書く。

4. GAP を書く。
   - TOBE と ASIS の差分を短く列挙する。
   - 差分が複数ある場合は、挙動、データ、API、設定、運用フロー、確認観点などに分ける。
   - GAP は対応内容の言い換えにしない。対応が必要になる理由を書く。

5. 対応方針を書く。
   - PR では、実際に変更した内容、変更しなかった内容、確認した内容を書く。
   - バックログでは、実施する作業、完了条件、除外範囲、確認方法を書く。
   - TOBE と GAP に対応する粒度で書く。

## PR 説明テンプレート

```markdown
## TOBE

- この PR により、...
- 対象プログラム / システムは、... できる状態になる。

## ASIS

- 現状は、...
- そのため、... という負荷 / リスク / 制約がある。

## GAP

- TOBE に対して、現状は ... になっている。
- ... が未整備なため、... が実行 / 検証できない。

## 対応内容

- ...
- ...

```

## バックログ / issue テンプレート

```markdown
## TOBE

- このバックログが完了すると、...
- 対象プログラム / システムは、... できる状態になる。

## ASIS

- 現状は、...
- 現在の制約 / 問題 / 未整備な点は、...

## GAP

- TOBE に対して、現状は ... になっている。
- この差分により、... が発生している / 実行できない / 検証できない。

## 対応方針

- ...
- ...

```

## 出力ルール

- ユーザーが別の形式を指定していない場合は Markdown で出力する。
- 見出しは原則として `TOBE`、`ASIS`、`GAP`、`対応内容` または `対応方針` を使う。
